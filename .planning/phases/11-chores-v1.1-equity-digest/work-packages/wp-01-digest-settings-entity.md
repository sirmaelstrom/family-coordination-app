## Implementation notes
- Entity (namespace `FamilyCoordinationApp.Data.Entities`): PK = `HouseholdId` (int, 1:1 with household).
  Fields: `string? WebhookUrlProtected` (ciphertext, nullable); `bool Enabled` (default false);
  `DigestCadence Cadence` (enum `{ Weekly }`, stored `HasConversion<string>()`, default `Weekly`);
  `DayOfWeek SendDayOfWeek` (default `Sunday`, stored as string or int — match codebase: store as `int`/enum
  string consistently with `ChoreEnums`); `int SendHourLocal` (0–23, default 18); `DateTime? LastSentAt`
  (UTC); `DateTime CreatedAt`/`DateTime UpdatedAt` (UTC). Household nav via `.WithMany()` no-arg (Category/
  Feedback pattern — do **not** add a column/nav to `Household.cs`, MN5/E2).
- Config: `HasKey(x => x.HouseholdId)` **and `builder.Property(x => x.HouseholdId).ValueGeneratedNever()`**
  (council — `HouseholdId` is the caller-supplied household FK/PK, NOT a generated identity; without this EF
  may treat a lone `int` PK as value-generated by convention and refuse caller-set values). Household FK
  `DeleteBehavior.Cascade` (matches RoomConfiguration); enum(s) `HasConversion<string>()` (codebase
  convention — applies to `Cadence`; store `SendDayOfWeek` as the BCL `DayOfWeek`, EF maps it int by default
  which is fine). The PK on `HouseholdId` **is** the index — do **not** add a separate index. Table name
  `"ChoreDigestSettings"`.
- DbSet: `public DbSet<HouseholdChoreDigestSettings> ChoreDigestSettings => Set<...>();`
- Migration: `dotnet build` **before** `migrations add` and **before** `migrations script` (WP-01 v1.0
  surprise: stale script otherwise). Confirm `.Designer.cs` emitted; `has-pending-model-changes` ⇒ none.
