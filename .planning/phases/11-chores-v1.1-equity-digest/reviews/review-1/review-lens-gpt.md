### Per-Package Findings

**WP-01: Digest-settings entity + migration**
- **Finding** (Severity: Major): `HouseholdId` is a single `int` primary key and EF may treat it as value-generated by convention unless explicitly disabled. That is wrong for a shared household FK/PK row where callers provide the household id.
  - **Location:** WP-01 Implementation notes; [ApplicationDbContext.cs](D:/Development/projects/family-coordination-app/src/FamilyCoordinationApp/Data/ApplicationDbContext.cs:6)
  - **Recommendation:** Require `builder.Property(x => x.HouseholdId).ValueGeneratedNever();` in the configuration. Also state the table name exactly as `"ChoreDigestSettings"`.

- **Finding** (Severity: Minor): The verification says the migration delta includes `CREATE TABLE "ChoreDigestSettings" (+ PK)` and constraints say `+ index`, but WP-01 also says the `HouseholdId` index is implicit as the PK. This is inconsistent.
  - **Location:** WP-01 Verification; constraints M13
  - **Recommendation:** Amend M13/WP-01 to say “single `CREATE TABLE` with PK/FK; no separate index required unless intentionally added.”

**WP-02: Equity calculator + DTO + contract fixture**
- **Finding** (Severity: Major): The DTO uses `equalSharePct` and `sharePct` as `double`, but the spec does not define whether these are fractions `0.25` or percentages `25.0`. WP-09 bars and fixture tests will drift.
  - **Location:** WP-02 Implementation notes; WP-09 EquityBoard notes
  - **Recommendation:** Define the wire contract explicitly: either rename to `share`/`equalShare` for `0..1`, or require `sharePct`/`equalSharePct` to be `0..100`. Update fixture expectations accordingly.

- **Finding** (Severity: Minor): “Week” boundary is only implied by the test example “Sun 23:59 vs Mon 00:01,” not declared as a rule.
  - **Location:** WP-02 Tests
  - **Recommendation:** State “week starts Monday in the configured chores timezone” or choose Sunday explicitly. Do not leave this to test inference.

**WP-03: DigestSettingsService**
- **Finding** (Severity: Major): Webhook clearing is underspecified. WP-03 says “empty string sentinel,” WP-11 says “a clear-sentinel,” and WP-06 only has `webhookUrl?`. Agents can easily implement incompatible clear behavior.
  - **Location:** WP-03 `UpdateAsync`; WP-06 Settings; WP-11 Implementation notes
  - **Recommendation:** Freeze the request shape: e.g. `webhookUrl: string | null | undefined`, where omitted means unchanged, nonblank means replace, and explicit `null` or `""` means clear. Use the same wording in WP-03/06/11.

- **Finding** (Severity: Major): Settings wire casing is not specified. With the existing global `JsonStringEnumConverter(JsonNamingPolicy.CamelCase)` in [Program.cs](D:/Development/projects/family-coordination-app/src/FamilyCoordinationApp/Program.cs:94), `DigestCadence.Weekly` and `DayOfWeek.Sunday` will serialize as `weekly`/`sunday`, but WP-11 suggests `cadence:'Weekly'`.
  - **Location:** WP-03 safe view; WP-06 request/response records; WP-11 update body
  - **Recommendation:** Define TS/backend contracts as `cadence: 'weekly'`, `sendDayOfWeek: 'sunday' | ...`, or intentionally use strings and validate them. Add a small settings DTO contract test or explicit fixture.

**WP-04: DigestBuilder + sender**
- **Finding** (Severity: Major): `DigestBuilder.Build(... IReadOnlyList<(string name, DueState)> choreDueness ...)` is awkward as a public contract and leaves the tuple element casing/meaning underdefined for WP-05. It also loses chore id/room/context, making later filtering/debug harder.
  - **Location:** WP-04 Implementation notes; WP-05 snapshot assembly
  - **Recommendation:** Define a named record, e.g. `DigestChoreLine(string Name, DueState DueState)`, and use `IReadOnlyList<DigestChoreLine>`.

**WP-05: DigestService.RunDueAsync**
- **Finding** (Severity: Critical): Idempotency is not concurrency-safe. The spec selects due rows, sends, then stamps `LastSentAt`. Two simultaneous cron requests can both pass the `LastSentAt` guard and double-post before either save commits.
  - **Location:** WP-05 RunDueAsync due/send/stamp flow; E10/M10 idempotency
  - **Recommendation:** Require an atomic claim step before send, or a transaction with row lock. For Postgres, specify `SELECT ... FOR UPDATE SKIP LOCKED` inside a transaction, or an atomic `UPDATE ... WHERE LastSentAt IS NULL OR LastSentAt < windowStart RETURNING ...`. Add an integration test that fires two `/digest/run` requests concurrently.

- **Finding** (Severity: Major): `GetDecryptedWebhookAsync` can return `null` on unprotect failure, but `IDigestSender.SendAsync` requires `string`. WP-05 does not say whether that household is skipped or failed.
  - **Location:** WP-03 unprotect behavior; WP-05 per-household send flow
  - **Recommendation:** Specify: if decrypted webhook is null, do not call sender; count as `Skipped` or `Failed` consistently, and log a warning without the value.

- **Finding** (Severity: Major): `DigestRunSummary { Sent, Skipped, Failed }` semantics are ambiguous. The query starts from enabled/configured settings rows, so disabled/unconfigured households are not naturally counted as skipped.
  - **Location:** WP-05 return summary; WP-06 endpoint response; WP-08 assertions
  - **Recommendation:** Define `Skipped` precisely, e.g. “enabled/configured but not due” only, or add separate `Disabled`/`Unconfigured` counts and adjust tests.

**WP-06: Endpoints + Program.cs wiring**
- **Finding** (Severity: Major): Equity and digest snapshot assembly will duplicate private `ChoreBoardService` logic because due/up-for-grabs helpers are private (`IsNeedsAttention`, `IsDueOrOverdue`) and no active-chore snapshot API exists.
  - **Location:** WP-06 Equity notes; WP-05 snapshot notes; [ChoreBoardService.cs](D:/Development/projects/family-coordination-app/src/FamilyCoordinationApp/Services/ChoreBoardService.cs:191)
  - **Recommendation:** Add an upstream/shared helper contract before WP-06, or allow WP-06/WP-05 to add a public `GetActiveChoreSnapshotAsync`/projection helper to `IChoreBoardService`. Otherwise parity with the board is easy to break.

- **Finding** (Severity: Major): The run endpoint placement is risky. `MapChoresEndpoints` currently creates an auth-required `/api/chores` group at [ChoresEndpoints.cs](D:/Development/projects/family-coordination-app/src/FamilyCoordinationApp/Endpoints/ChoresEndpoints.cs:25). Adding `/digest/run` inside that group violates the cron-token design.
  - **Location:** WP-06 Run trigger
  - **Recommendation:** Specify exact implementation: add an `app.MapPost("/api/chores/digest/run", ...)` route in `MapChoresEndpoints` before/after the auth group but not on `group`, or map it in `Program.cs` before `app.MapChoresEndpoints()`.

- **Finding** (Severity: Minor): `.env.example` may not be the only env documentation. The orchestrator says deploy note + morning handoff, but no WP owns those files.
  - **Location:** WP-06 Files; E19
  - **Recommendation:** Add the actual deploy note/handoff file path to WP-06 or a tiny final ops-doc WP.

**WP-08: Integration tests**
- **Finding** (Severity: Major): The fake sender failure test says “throws for Household A only,” but `IDigestSender.SendAsync` receives only `webhookUrl` and `DigestModel`, not `householdId`.
  - **Location:** WP-08 Failure isolation
  - **Recommendation:** Tell the test to distinguish by seeded webhook URL or by `DigestModel` headline/household name. Do not imply household id is available at the sender boundary.

- **Finding** (Severity: Major): Existing WP-08 tests will not catch concurrent double-fire idempotency, which is the real failure mode for external cron.
  - **Location:** WP-08 Idempotency
  - **Recommendation:** Add a concurrent run test: issue two valid `POST /api/chores/digest/run` calls at the same fixed instant and assert one send per household total.

**WP-09: Island Equity lens**
- **Finding** (Severity: Major): Equity cache invalidation is missing. The store fetches equity once on first open and on window switch, but completing chores changes `ChoreCompletion`; seed/backfill also changes counts. The equity lens can become stale during the same session.
  - **Location:** WP-09 Store notes; [state.svelte.ts](D:/Development/projects/family-coordination-app/frontend/chores/src/lib/state.svelte.ts:404)
  - **Recommendation:** Require `boardStore.complete`, `seedStarter`, and any board refetch that changes completions to invalidate/reload equity if loaded. Alternatively do not cache equity across lens openings.

- **Finding** (Severity: Minor): “loadEquity fetches once when the lens is first opened” is not mechanically tied to the existing `setLens` path, which is synchronous.
  - **Location:** WP-09 state/App routing
  - **Recommendation:** Specify either `setLens` triggers a fire-and-forget `loadEquity`, or `App.svelte` has a `$effect` watching `store.lens === 'equity'`.

### Cross-Cutting Findings

- **Finding** (Severity: Critical): The digest idempotency guarantee is stronger than the implementation plan. “At most once per cadence window” cannot be satisfied by a read-send-update loop under concurrent requests.
  - **Recommendation:** Add DB-level atomic due claiming and a concurrent integration test before dispatch.

- **Finding** (Severity: Major): Settings contracts lack a fixture or exact JSON contract. Unlike board/equity, settings have no contract test, yet they carry enum casing, secret masking, and clear semantics.
  - **Recommendation:** Add a small `digest-settings.json` fixture or explicit DTO contract test in WP-03/WP-06.

- **Finding** (Severity: Major): Shared chore dueness/up-for-grabs logic is not exposed. WP-05/WP-06 will duplicate private board logic and can drift from the island board.
  - **Recommendation:** Create or expose a shared projection/snapshot helper before the endpoint and digest WPs.

### Targeted Analysis Results

**A. Self-Containment Audit**
- WP-11 depending on WP-06 is not self-contained enough. Reading WP-11 alone, an agent cannot know the exact wire values for `cadence` and `sendDayOfWeek`, nor the clear-webhook sentinel.
- WP-10 depending on WP-06/WP-09 is also incomplete. It says edit photo, but does not tell the agent to add `photoPath` to the frontend update request or use the upload endpoint before PUT.

**B. Interface Contract Check**
- Traced settings flow: `HouseholdChoreDigestSettings.Cadence` / `SendDayOfWeek` → `DigestSettingsView` → `GET /api/chores/digest-settings` → `DigestSettingsView` TS → `PUT /digest-settings`.
- Mismatch: backend enum DTOs will normally serialize as camelCase strings (`weekly`, `sunday`) under global JSON options, while WP-11 suggests `Weekly`. Clear webhook is also inconsistent: omitted vs empty string vs unspecified sentinel.

**C. Blast Radius Scan**
- WP-10 Files are incomplete: all lens components that pass props to `ChoreCard` must change.
- WP-06/WP-05 likely need a shared service/helper beyond their listed files if the spec wants parity with board dueness/up-for-grabs logic.
- WP-06 docs are incomplete for E19: `.env.example` is listed, but deploy-note/handoff ownership is not.

**D. Wave Ordering Challenge**
- Move the shared chore snapshot/projection helper earlier, either into WP-02 or a new WP before WP-04/05/06. That reduces duplication risk.
- Move settings contract finalization into WP-03 and make WP-11 consume a frozen fixture/TS shape.
- Keep WP-10 after WP-09, but amend its blast radius before dispatch. Current sequencing is fine; current file ownership is not.

### Overall Assessment

**Verdict:** Needs amendments

The spec is close structurally, but not ready for autonomous dispatch. The main blockers are concurrency-safe digest idempotency, incomplete frontend edit/photo blast radius, ambiguous settings JSON contracts, and duplicated board dueness logic. Fix those before sending agents; otherwise the likely outcome is green unit tests with stale UI, casing drift, or a cron double-post bug that only shows up in production.
